# 本地源码安全隔离 — 场景决策参考

## 什么时候需要先问用户？

| 场景 | 行为 |
|------|------|
| 用户说"分析一下这个目录"（指向 LI 项目路径） | **先问**：路径 + "这些是 LI 领域源码，是否确认让我读取？" |
| 用户直接贴了一段 LI 相关代码 | **先问**："这段代码看起来是 LI 领域的，确认让我分析？" |
| 用户描述了一个 LI 相关的 bug 现象 | **先问**："需要我查一下相关源码定位问题吗？" |
| 用户明确说"改一下 XXX 文件的 YYY 函数" | 可执行（用户已明确授权），但只做用户指定的操作 |
| 用户说"帮我 review 这个 PR/commit"（涉及 LI 代码） | **先问**："确认让我查看这些代码差异？" |
| 用户说"查一下这个 ASN.1 类型的定义" | 如果 ASN.1 文件在 ETSI-ASN1-Assistant 项目下，**先问** |
| 用户问公共标准（3GPP/ETSI RFC）的内容 | 直接回答，不需问（属于 LEVEL 1 公开知识） |

## 已确认可操作后的限制

即使获得用户明确许可，仍然禁止：
- 将源码片段粘贴到 `web_search` 搜索
- 将源码发送到 `web_extract` / `browser` 等联网工具
- 将源码细节通过 `text_to_speech` 语音播报（内容泄露）
- 将源码结构描述发给在线 LLM 用于"重构建议"

## 例外场景：用户明确说"帮我操作"

当用户说"帮我操作"时，可执行以下本地操作而不需要额外确认：
- `read_file` / `search_files` 查看文件内容
- `patch` / `write_file` 修改代码
- `terminal` 运行编译/测试命令
- `terminal` 执行 git 操作

但依然禁止调用任何联网或在线 LLM 工具处理源码内容。

## 禁止误判场景（这些不需要问）

以下场景不触发源码安全提示：
- 用户问 ETSI TS 102 232 标准中的 X2 接口定义（公共标准）
- 用户问 3GPP TS 23.003 中的 IMSI 格式定义（公开规范）
- 用户问通用的 ASN.1 语法或 BER/DER/PER 编解码原理
- 用户问 Linux 系统命令或 Python 语言特性

## 项目目录扩展规则

新增 LI 相关项目后，在 `SKILL.md` 的覆盖范围表中添加：
1. 确认项目路径和内容类型
2. 添加到 LEVEL 6 覆盖范围表
3. 如果项目不在 `~/work-projects/` 下，同步更新核心禁令中的目录列表

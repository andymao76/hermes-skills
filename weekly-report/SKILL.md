# 运维开发周报

生成适用于运维/开发人员的专业周报。

## 核心公式

本周关键成果（数据对比）+ 问题复盘 + 下周规划

## 撰写原则

1. **成果先行**：开头甩3个最重要的完成事项
2. **价值导向**：做了什么 + 带来什么效果
3. **问题显价值**：遇到什么问题→怎么排查→结果如何
4. **计划具体化**：做什么、做到哪一步、预期目标
5. **数据化表达**：性能提升X%、耗时缩短Y%、覆盖率Z%

### 数据收集策略

周报数据来源于四个渠道，按优先级排列：

1. **日报文件**（最高优先级）— `~/sinovatio/日报/maohengzhen_日报_YYYYMMDD.docx` 或 `.md`
   - 也检查归档目录：`~/work-projects/_archive/sinovatio/日报/`
2. **Brain log** — `~/Documents/Obsidian Vault/Brain/log/YYYY-MM-DD.*.md`
3. **session_search** — 会话级别的讨论、调研、代码开发、cron 作业执行记录
4. **项目文件** — `~/Documents/Obsidian Vault/工作/*`

#### ⚠️ 常见陷阱：数据稀疏

**场景**：本周只有少数几天有日报，甚至没有日报。Brain log 目录也可能为空。

**处理方法**：
- 不要只依赖日报文件就写周报——大量工作（LI 技术研究、SoapUI 配置、系统排障）只存在于会话记录中
- 使用 session_search 按关键词分轮搜索，覆盖所有项目领域：
  1. `sinovatio A1 印尼 埃及 老挝 菲律宾 工作` — 获取项目相关活动
  2. `Ericsson SoapUI E-LIMS 安装` — 技术研究类活动
  3. `肯尼亚 SMATS OWLS ZTLIG LIG` — 特定项目/方案文档
  4. `cron 安全审计 系统架构 备份 迁移` — 系统运维活动
- cron 作业的 session 数据（健康检查报告、安全审计摘要、IMA 同步报告）可直接提取关键指标填充系统运维章节
- 每轮 session_search 的返回结果含 bookend_start 和 ±5 消息窗口，足以重建上下文。需要更多细节时用 session_search(session_id, around_message_id) 滚动查看

**搜索关键词速查**：
| 项目领域 | session_search 关键词 |
|----------|----------------------|
| A1 PC项目 | `A1 苏丹 ZTLIG OWLS SICMS` |
| 印尼项目 | `印尼 ASIOC` |
| 埃及项目 | `埃及 NSS 车载定位` |
| 老挝项目 | `老挝 MOPS IFS` |
| 菲律宾项目 | `菲律宾 AFFRS100A` |
| LI 技术研究 | `5G LI 合法拦截 Ericsson SoapUI 华为` |
| 系统运维 | `cron 安全审计 备份 部署 docker 巡检` |

### 生成步骤

1. **搜数据**：先 `ls -la ~/sinovatio/日报/` 检查日报覆盖哪些天，再按上述策略用 session_search 补全缺失信息
2. **写 Markdown 版**：按下方模板组织内容，写入 `~/sinovatio/周报/maohengzhen_周报_YYYYMMDD-YYYYMMDD.md`
3. **生成 DOCX 版**：使用 python-docx 生成格式化的 Word 文档。参考脚本模板见 `templates/gen_weekly_docx.py`
4. **验证**：确认两个文件均正确写入，检查文件大小


## 岗位关注点

### 运维
| 类型 | 核心指标 |
|------|----------|
| 前线运维 | 日均告警数↓X%、MTTR↓Y% |
| SRE/平台 | 自动化占比↑X%、节省人力Y小时 |
| 架构/稳定性 | 服务可用性99.X%、QPS↑Y% |
| 安全合规 | 漏洞修复率X% |

### 开发
| 关注点 | 量化示例 |
|--------|----------|
| 功能开发完成度 | 完成订单模块开发（100%），已提测 |
| 性能优化 | 接口P99从2.1s降至0.8s，超时率↓87% |
| Bug修复 | 修复3个线上Bug |
| 需求沟通 | 参加需求评审X场 |

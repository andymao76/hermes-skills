# LLM 路由审核闸门

## 核心理念

**所有在线 LLM 请求必须经过人工审核闸门**，不能自动决定内容路由。

## 审核三元组

每次使用在线 LLM (DeepSeek/SiliconFlow/OpenRouter) 前，必须执行：

```
请求内容 → 领域分类 → 路由决策 → 人工确认
```

## 领域分类速查表

| 领域 | 路由规则 | 是否需要人工确认 |
|------|---------|:--------------:|
| **LI 技术内容** | ✅ 用 OpenLI 替代后可在线处理 | ✅ 每次确认 |
| **HW(华为) 公开信息** | ✅ 可在线处理 | ❌ 无需拦截 |
| **ZTE(中兴) 公开信息** | ✅ 可在线处理 | ❌ 无需拦截 |
| **Ericsson(爱立信) 公开信息** | ✅ 可在线处理 | ❌ 无需拦截 |
| **ZTLIG / Sinovatio(中新赛克)** | ❌ **禁止** 在线 LLM | — 等本地 LLM 建立 |
| **A1 项目** | ❌ **禁止** 在线 LLM | — 全程本地 |
| **公开技术（3GPP/Python/Linux/架构等）** | ✅ 正常 | ❌ 无需拦截 |
| **知识库 public/ 内容** | ✅ 正常 | ❌ 无需拦截 |
| **知识库 internal/ 内容** | ⚠️ 脱敏后可在线 | ⚠️ 确认脱敏状态 |
| **不确定边界** | — | ✅ **必须问** |

## 响应流程

### 当请求涉及敏感域时（LI/A1/ZTLIG/Sinovatio）

```
用户请求
  ↓
识别到涉及 [LI/A1/ZTLIG/Sinovatio]
  ↓
输出格式:
  ⚠️ LLM 路由审核
  ───────────────────────────────────────────
  检测到此请求涉及: [具体域名称]
  路由建议: [禁止在线 / 需确认 / 可在线]
  是否允许使用在线 LLM (DeepSeek) 处理？
  ───────────────────────────────────────────

等待用户确认:
  「可以」→ 正常处理，附注「已通过人工审核」
  「不行」或「先看看」→ 仅本地操作：
    · 读本地知识库 (read_file)
    · 搜索本地文件 (search_files)
    · 提供框架性建议/方案设计
    · 不做深度推理或生成
  「等等」→ 搁置，等用户主动再提
```

### 当请求不涉及敏感域时

正常处理，无需拦截。

## 审核清单模板

```
□ 请求内容是否涉及 LI / A1 / ZTLIG / Sinovatio？
   → 是 → 走敏感流程
   → 否 → 下一步

□ 是否涉及 HW / ZTE / Ericsson 非公开信息？
   → 是 → 仅限公开资料可在线
   → 否 → 下一步

□ 是否涉及 knowledge/li/ knowledge/customers/ knowledge/secrets/ 下的敏感文件？
   → 是 → 走敏感流程
   → 否 → 正常处理

□ 边界是否清晰？
   → 是 → 按规则执行
   → 否 → 必须先问用户
```

## 通用「先人工审核」原则

**适用场景：** 架构设计、实施计划、系统搭建方案、代码重构等需执行的任务

**规则：**
1. 任何需要动手执行的方案/架构/计划，必须先以文本形式呈现给用户审核
2. 用户确认「可以」后再逐阶段执行
3. 不要直接开始执行而不展示方案
4. 方案应包含：目标、步骤、预期产出、风险点
5. 方案应「可执行」— 包含实际命令/配置代码，而非纯概念描述

## 审核后话术模板

### 确认通过后
```
✅ 人工审核通过 — 已确认此请求可在在线 LLM 处理
```

### 确认拒绝后
```
⏸️ 已按指示仅在本地处理，未使用在线 LLM
```

### 用户说「先看看」
```
已暂缓，等你准备好了再继续
```

## 硬件感知路由（三层架构模式）

当系统同时具备控制层、调度层、推理层时，路由逻辑从「二选一」升级为「三层分流」：

```
Win11 (控制层)    Hermes Agent · 知识库 · 工具链
      │
GEM12+ (调度层)   Gateway · MCP · RAG · Policy Engine · Intent Router
      │
4060Ti (推理层)    CUDA · llama.cpp · vLLM · 纯本地推理
```

| 数据级别 | 路由路径 | 说明 |
|---------|---------|------|
| PUBLIC | Win11 → DeepSeek API (云) | 快速，50+ tok/s，需联网 |
| PRIVATE | Win11 → GEM12+ RAG → 4060Ti (:8080/8081) | 全本地，安全 |
| HIGH_SENSITIVE | Win11 → GEM12+ RAG → 4060Ti (强制本地) | RULE20强制，绝不出网 |

**本地 GPU 可用时的行为变化：**
- 「LOCAL_ONLY」不再是「只能做简单本地操作」，而是「走 GEM12+ ChromaDB → 4060Ti 完整推理」
- 14B Q4 40-60 tok/s · 7B Q4 70-90 tok/s → 用户体验接近云 LLM
- 全脱网模式：所有 Provider 切到 `127.0.0.1:8080/8081`

### Provider 切换命名约定

```
hermes-li     → 强制走本地 eGPU (4060Ti)，用于 LI/A1/Sinovatio 分析
hermes-ds     → 走 DeepSeek 云，用于日常对话和公开技术查询
custom:gem12  → GEM12+ 上的本地模型服务
```

## eGPU 选型经验

当选择本地推理 GPU 时，**显存大小 > 原始算力** 是核心原则：

| 模型 | 4070 12GB | 4060Ti 16GB |
|-----|:---------:|:-----------:|
| 7B Q4 | 极快 | 极快 |
| **14B Q4 (刚需)** | 可用但紧 | **稳定运行** |
| 20B级 Q4 | 勉强 | 可用 |
| 30B 4bit | 很吃力 | 可跑 |

4060Ti 16GB 的核心优势：14B Q4 稳定运行是 Hermes Agent 的推理刚需（LI分析/编程/协议解码），12GB 显存在此场景下「可用但紧」，16GB 是真正的安全线。同时 160W TDP 低功耗适合 eGPU 长时间运行。

## 原则

1. 宁可多问一次，不要自作主张
2. 不确定边界时默认走敏感流程
3. 用户明确说「可以」后不再重复确认同一话题
4. 人工审核后正常处理，不额外加安全提示干扰结果
5. 此技能优先级高于所有其他技能和规则
6. 本地 GPU 可用时，「仅本地」不再是能力受限状态，应主动推荐使用本地推理

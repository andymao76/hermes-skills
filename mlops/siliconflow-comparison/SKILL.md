# SiliconFlow 国内版 vs 国际版 API 对比

## 核心对比快览
| 维度 | 国内版 (api.siliconflow.cn) | 国际版 (api.siliconflow.com) |
|------|-----------------------------|------------------------------|
| **首字延迟 (TTFB)** | 国内极快（通常 < 200ms） | 海外极快 / 国内需看跨境网络（通常 500ms+） |
| **最佳调用客户端** | 部署在国内（阿里云/腾讯云等）的服务器 | 部署在海外（Vercel / Cloudflare / AWS 海外）的服务器 |
| **网络路由** | 走国内 BGP 直连，无墙，无跨境抖动 | 走海外 CDN / 边缘节点，受跨境网络波动影响大 |
| **限流策略 (Rate Limit)** | 免费档 TPM 限制较严格，高并发易报 HTTP 429 | 与国内版独立计费，部分独占模型限流政策不同 |

## 详细时延分析

### 1. 物理网络延迟（RTT）
- **`.cn` 域名**：服务器主要托管在国内主流云厂商机房（如阿里云、腾讯云）。在中国大陆发起请求，Ping 值通常在 10ms - 50ms 之间。
- **`.com` 域名**：服务器和接入点主要面向海外用户。从中国大陆直接调用 `.com`，流量需要经过跨境网关，Ping 值通常会飙升至 150ms - 300ms 以上，晚高峰极易出现丢包或连接超时。

### 2. 模型推理延迟（生成速度）
两边的底层算力集群有隔离，但均基于硅基流动自研的加速推理引擎，纯模型推理速度（Tokens/秒）基本持平。
- **关键差异**：大模型流式传输（Streaming）是逐字返回的，**首字延迟（TTFB）**受网络影响极大。在国内环境调用 `.cn` 会让人感觉“秒开”；调用 `.com` 则会有明显的“吸一口气再吐字”的停顿感。

### 3. 导致“高时延”假象的原因：速率限制 (Rate Limit)
- `HTTP 429: TPM limit reached` 属于业务级限流。
- 当应用并发稍高，或测试代码未加请求间隔（短时间内多次重试），触碰频率天花板，服务器会直接秒回 429。
- 框架中频繁的 429 被捕获并进行指数退避重试（如等待 2.4s、4.2s），在外部表现起来就像是 "API 响应极慢/死锁"。

## 最佳实践建议

1. **根据服务器位置选域名**：
   - 业务代码跑在**国内服务器** -> 必须使用 `https://api.siliconflow.cn/v1`
   - 跑在**海外云**或 Nextchat/LobeChat 等海外托管面板 -> 优先使用 `https://api.siliconflow.com/v1`
2. **解决 429 限流问题**：
   - 在代码中加入**请求队列**或**限流器（Rate Limiter）**，限制每分钟的 Token 消耗。
   - 如果是免费额度，考虑升级为 Pro（付费）账户，付费用户的 TPM 和 RPM 阈值会大幅提升。

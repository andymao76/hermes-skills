# Document Synthesis — 多源文档综合

从多个本地文件合成一份结构化综合文档的工作流。

## 触发条件

当用户要求：整合、综合、合并、汇总多个文档（PDF/DOCX/MD/TXT）为一个设计文档或技术文档时触发。
典型句式：*"把这些整合成一份综合设计文档"* / *"把 XX 和 YY 整合起来"* / *"综合这几个文档"*

## 工作流

### 1. 源文件清单

列出所有源文件路径和格式。验证文件可读。重点关注：
- 格式兼容性（PDF→pdftotext，DOCX→python-docx，MD/TXT→直接读取）
- 文件大小（30K+ 字的大文件先完整阅读，小文件随读随用）
- 时效性（同一主题的旧版/新版需标注版本差异）

### 2. 内容提取

按格式选择提取方法，结果存 `/tmp/`：

| 格式 | 提取命令 | 注意 |
|------|---------|------|
| `.pdf` | `pdftotext "$src" /tmp/out.txt` | 表格可能丢失结构；之后 `wc -c` 验证 |
| `.docx` | `python3 -c "from docx import Document; ... for p in doc.paragraphs: print(p.text)"` | 跳过空段落 |
| `.md` / `.txt` | `read_file path` | 原生格式，直接可读 |
| `.html` | `lynx -dump -nolist` 或 `pandoc -f html -t markdown` | 剥离标签 |

### 3. 内容分析

逐一阅读所有源文件后，按三轴分析：

| 维度 | 问题 | 决策 |
|------|------|------|
| **唯一性** | 哪些内容只出现在这一个源？ | 必须保留，否则丢失 |
| **重叠** | 哪些内容出现在多个源？ | 去重或交叉引用，优先用最完整版本 |
| **空白** | 哪些主题没有任何源覆盖？ | 标注为待办/未来工作 |

### 4. 结构设计

综合设计文档的标准交付结构：

```
integrated-design-v1/
├── README.md           ← 主文档（综合设计）
├── dashboards/         ← 可视化仪表盘索引
│   └── README.md
├── templates/          ← 可复用配置模板
│   ├── docker-compose.yml
│   └── exporter-config.yaml
└── scripts/            ← 自动化脚本
    └── healthcheck.sh
```

纯文档项目（无附属物）可简化为仅 `README.md`。

### 5. 综合撰写

主文档撰写核心规则：

1. **按主题组织，不按来源组织** — 每个章节写"ZTLIG 架构"而非"从 ops_monitor 来的内容"
2. **每章标注来源** — 章首或章尾标注"本节主要参考：xxx.pdf 第N章"
3. **保留唯一内容** — 某个源独有的细节必须保留，不要因为"其他文档没提"就丢掉
4. **冲突处理** — 多个源对同一主题有不同说法时，列出各方观点并说明采纳了哪个
5. **完整覆盖** — 确保每个源的核心内容都在输出中有对应章节

### 6. 附属文件

- **dashboards/** — Grafana 仪表盘设计说明（面板数量、目标用户、指标清单），非 JSON 原始文件
- **templates/** — 可复制使用的配置（docker-compose / exporter YAML / 环境变量示例）
- **scripts/** — 可执行的自动化（健康检查 / 批量处理 / 数据采集），带 `chmod +x`

### 7. 来源对照表

文档末尾必须包含来源对照表，精确到文件：

```markdown
## 来源对照

| 来源 | 格式 | 字数 | 章节引用 | 内容说明 |
|------|------|------|----------|----------|
| ops_monitor_design_v3.pdf | PDF | ~33K | 第1-10章 | 主要架构参考 |
| AIOps重设计V3.docx | DOCX | ~3.2K | 第5.1节 | AIOps概念集成 |
| A1架构分析.txt | TXT | ~9.5K | 第3章 | 拓扑+进程清单 |
| li-platform-monitoring.md | MD | ~16.8K | 第2/4章 | 监控指标 |
```

## 陷阱

- **PDF 表格丢失** — pdftotext 后表格数据可能乱行，关键表格（如进程列表、端口映射）需手动验证
- **大文件耗上下文** — 30K+ 字的文件先完整读取再写，写到具体章节时除非需要核对细节，不要重复读取
- **不要静默丢弃唯一内容** — 如果某个细节只存在于一个源，它就是不可替代的
- **不要创建空子目录** — 只在有内容要放时才建 dashboards/ / templates/ / scripts/，空目录让用户困惑
- **来源标注必须具体** — 写"来自本地知识库"不够，必须写文件路径、格式和内容范围
- **不要按来源分章节** — 最自然的写法是按主题（架构→监控→巡检→模板），而不是按来源（文件A→文件B→文件C）

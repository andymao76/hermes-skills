# 分段文档分享模式（Incremental Document Ingest）

## 概述

用户通过分段粘贴的方式分享技术文档，期望知识库条目在此过程中逐步构建并增量更新。

**两种粒度模式**：

| 粒度 | 描述 | 示例 |
|------|------|------|
| **段落级** | 用户每次发一个完整段落/章节 | "学习 ZTE LI HI1"，然后发整段文档 |
| **参数级（超细粒度）** | 用户每次只发一个参数名或一行字段 | `"Description"` → `"Syntax"` → `"tno_id M Integer"` → `"doo M Date of order"` |

**典型表现形式**：用户先发"学习 ZTE LI"，再发"学习 ZTE CS HI2"，再发具体段落（"Arguments"、"Status"、"0"、"201,202"）逐段提示递进。

## 工作流

### 阶段 1：识别模式

| 信号 | 含义 |
|------|------|
| 用户说**"学习 [主题]"** | 新内容开始，准备分析入库 |
| 用户发**单个词或短语**（"Arguments", "Status", "Syntax", "Remarks"） | 提示当前位置，继续分享文档 |
| 用户发**单独数字或短字符串**（"0", "201,202", "Value"） | 补充具体参数/状态码值 |
| 用户说**"好的" / "OK"** | 确认入库请求，执行写入 |

### 阶段 2：逐段处理

每次收到新内容时：

1. **识别上下文**：根据之前分享的内容判断当前属于哪个章节
2. **分析新内容**：解析参数定义、状态码、业务规则等
3. **增量更新**：在知识库文件中追加或修改对应章节

### 阶段 3：超细粒度参数级分享模式

当用户以**参数级粒度**（每次只发一个参数名+描述，甚至只发一个单词）分享内容时，按以下流程处理：

#### 累积原则

| 做法 | 理由 |
|------|------|
| ✅ **在内存中累积参数**，等待自然断点再 patch | 避免每收到一个参数就调一次 patch（降低 token 开销，减少文件冲突风险） |
| ✅ **自然断点**：Status / Output / Examples / 用户沉默 / 用户切换话题 | 这些标志一个功能块（如 tdel 命令）的完整描述到达尾声 |
| ✅ **断点到达后，一次性 patch 整个功能块** | 将之前累积的所有参数+状态码+描述合并为一次 patch |
| ❌ **每收到一个参数名就调 patch** | 大量冗余操作，可能相互覆盖 |

#### 具体步骤

```
收到 "Description"
  → 内存记录: tmod.Description = "modify an existing target"
  → 回复确认，等待继续

收到 "tno_id M The target number Id. Format is integer."
  → 内存累积: tmod.params = {tno_id: {M, "目标唯一ID"}}
  → 回复确认，等待继续

...（累积10+个参数）...

收到 "Status"
  → 识别到自然断点
  → 将之前累积的所有参数合并为一次 patch 写入知识库
  → 回复确认

收到 "0" → 累积: tmod.status = {0: "成功"}
收到 "201,202" → 累积: tmod.status += {201, 202}

收到 "Output" → 自然断点到达 → 一次性 patch 所有状态码
```

#### 判断自然断点的优先级

| 断点信号 | 处理方式 |
|---------|---------|
| Status / Output / Examples | 立即 batch patch 之前累积的全部内容 |
| 用户说"继续"或否认更多内容 | 说明当前块未结束，继续累积 |
| 用户主动切换话题（"学习 [新主题]"） | 先 batch patch 未决内容，再处理新话题 |
| 超过 10 个参数未 patch | 考虑主动触发 batch patch（避免内存累积过大丢失）|

## 最终入库准则

| 准则 | 说明 |
|------|------|
| 首条写入 | 首次确认时用 `write_file` 创建完整结构框架 |
| 后续更新 | 一律用 `patch` 在已有笔记的对应章节追加 |
| 保留占位 | 尚未覆盖的章节标记 `(待补充)` |
| 结构稳定 | 首版就确定好章节结构，后续只补充不重构 |
| 交叉链接 | 首版就建好 `[[wikilinks]]`，后续新增内容无需改链接 |
| 索引更新 | 仅首次写入后需要（后续 patch 不影响索引）|
| **批量 patch** | 积累到自然断点后一次性 patch，不逐行操作 |
| **厂商优先分类** | LI 文档归属优先级：厂商名 > 项目名 > 通用目录。Ericsson/华为/ZTE 的文档先往 `li/<vendor>/` 放，不要默认放 `li/projects/` 或 `research/` |

## 示例流程

```
用户: 学习 ZTE LI HI1
Agent: 分析并总结 → "是否写入知识库？"
用户: OK
Agent: write_file 创建 ZTE_CS_LI_HI1_HI2_HI3_三接口规范.md

用户: tdel tno_id=TargetNoId doo=DateOfOrder
Agent: 分析参数 → patch 更新 tdel 章节

用户: Arguments
Agent: 等待继续分享

用户: Status
Agent: 等待继续

用户: 0
用户: 201,202
Agent: patch 更新状态码表

用户: doo M Date of order.
Agent: patch 补充 doo 描述
```

```

用户: Description

用户: liid=LiId net=Networks dtype=DataTypes

  → 等待状态码继续

用户: 0
用户: 201, 202

  ...（累积状态码，不 patch）...

```

## 区别于口述采集

| 维度 | 口述采集 | 分段文档分享 |
|------|---------|------------|
| 内容来源 | 用户用自然语言口述知识 | 用户粘贴文档原文 |
| 用户提示 | 用句子描述概念 | 用关键词/段落标识进度 |
| Agent 工作 | 结构化编排自然语言 | 直接解析原文 + 编表 |
| 更新方式 | 随行编译后展示 | 在断点处 batch patch |
| 校验方式 | 让用户确认概念准确性 | 按文档原文录入 |

## 关键注意事项

### 1. 原文措辞必须精确

用户会逐字校对英文措辞，即使拼写错误（如文档原文写 `accurancy` 而非标准的 `accuracy`）也必须**按原文录入**，不要自作主张修正。常见需要精确匹配的场景：

| 场景 | 原文如此 | 不要改成 |
|------|---------|---------|
| 参数描述 | "The monitoring center ID for call content (voice)" | 缩写为 "语音 MC ID" |
| 状态码含义 | "LIID 冲突" | "LIID 已存在" |
| 拼写差异 | "configuered"（文档原文） | "configured"（修正拼写）|
| 术语差异 | "horizontal accurancy"（文档原文） | "horizontal accuracy"（标准写法）|
| 标点差异 | "IN, OUT oder BOTH"（文档原文德语式）| "IN, OUT or BOTH" |
| 格式标注 | "(Format is integer, unit is meter)" | 省略括号内容 |
| 条件标注 | "C" (conditional) 与 "O" (optional) 有根本区别 | 混用 M/O/C 符号 |

**例外**：如果明显是拼写错误且影响理解，先在笔记中保留原文措辞，然后在括号中标注说明，如 `configuered（原文如此）`。

### 2. 单字符和空信号处理

用户可能发空字符串或单字符（如 `?`, `(`, `)`）表示正在继续同一段描述的中间部分。此时：

| 信号 | 含义 |
|------|------|
| `?` | 等我继续发或确认你已跟上 |
| `)` | 上一个参数的描述括号结尾 |
| `(cf. 4.5.2)` | 补充文档引用，追加到上一个参数描述 |
| 空消息 | 继续等待，不要催促或跳步 |

### 3. M/O/C 标注一致性

参数表的 M/O/C 列在整个文档中是一致的：
- `M` = Mandatory（必填）
- `O` = Optional（可选）
- `C` = Conditional（条件可选，仅在特定网络/MC类型下可用）

不要在中间某处用缩写如"同 mc_voice 语义"代替原文描述，用户会要求逐条写全。

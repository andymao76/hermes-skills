# Source Code Study — 源码学习与知识提取

## 触发条件

当用户要求"学习"、"分析"、"研究"、"看下"某个目录或文件中的源代码时触发。

- `学习 /path/to/source/`
- `分析这个文件`
- `看下这个代码`

## 工作流

### 阶段一：探查

1. **列出目录结构** — 了解文件数量和类型
2. **判断文件格式** — 文本 / zip / 压缩包
3. **评估大小** — 大文件先读头尾，小文件全量读

### 阶段二：阅读与理解

4. **按依赖关系阅读** — 头文件 → 实现文件
5. **识别架构**：
   - 模块划分、函数分组、文件职责
   - 数据流（输入→处理→输出）
   - 接口层 vs 实现层
6. **记录调用链** — 核心入口和出口

### 阶段三：Bug 排查

7. **检查常见 Bug 模式**：
   - `memset` / `memcpy` 大小参数
   - `sizeof(指针)` 误用
   - API 返回值未检查
   - 空指针 / 双指针解引用错误
   - 循环条件错位（`for(i=0; i++; i<N)`）
   - 类型截断（`size_t` → `UINT32`）
   - `fread` 不足读取未处理

### 阶段四：修复

8. **最小化修改** — 只改 Bug，不重构风格
9. **保持一致性** — 参考同文件中同类函数的处理方式
10. **验证** — 读回确认修复逻辑

### 阶段五：知识沉淀

11. **写知识库**：
    - 文件概览、架构、函数分类
    - Bug 清单及修复记录
12. **更新索引** — 上级 `_index.md` 加链接

## 陷阱

- `sizeof(指针)` ≠ `sizeof(结构体)` — 除非是数组名
- 二维数组 memset 只传一维只清第一行
- `fread` 用 `if(!blen)` 不捕获部分读取
- 注释掉的代码块可能是废弃逻辑
- 同文件同类函数都检查了返回值唯独某个没检查 → 就是 Bug

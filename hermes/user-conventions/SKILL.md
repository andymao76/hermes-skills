# 用户操作约定与偏好

## 三层记忆系统（RULE / MEMORY / SKILL 区分）

用户将 Hermes 的记忆系统明确分为三层：

| 层 | 用途 | 问什么 | 内容标准 | 配额 |
|----|------|--------|----------|------|
| **user profile** | WHO | 我是谁 | 身份、角色、核心能力、领域 | ~15% |
| **memory** | WHAT | 当前有什么 | 紧凑核心事实、状态、项目分类规则 | ~10% |
| **skills** | HOW | 怎么做 | 可复用流程、操作约定、工作流 | 不限 |

关键区分原则：
- 用户画像 = 身份，不存操作规则或工作流
- Memory = 一句话能说完的核心事实，不存多步流程
- Skill = 怎么做一件事的完整流程和操作步骤
- 规则类（铁律、禁止 sudo）放 operational-playbook

### Skill 正确性验证机制

用户通过 skill-creator 的标准化流程验证技能质量：

```
写 SKILL.md → 创建测试用例 (evals/) → 并行跑 with-skill + baseline
→ 定量断言评分 → Viewer 对比输出 → 用户反馈 → 迭代改进
```

此外通过进化闭环持续改进：问题解决 → 总结经验 → 写入 Knowledge → 提炼 Skill → curator 自动清理。
具体详见 `skill-creator` 和 `hermes-evolution-mechanism` 技能。

### 版本历史记录规则

项目迭代时必须记录版本变更历史。要求：
- 每次版本变更在项目 `docs/` 下创建或追加 `VERSION_HISTORY.md`
- 格式：`## Vx.y (YYYY-MM-DD)` 三级标题
- 分三部分：**新增** / **变更** / **修复**（可选）
- 每个版本包含日期、主要变更摘要、影响范围
- 代码中显示版本号的位置（导航栏标题、页脚）同时更新
- **版本升级后必须全面检查并更新所有项目文档**，包括 README、系统设计文档、架构文档等，不能只更新 VERSION_HISTORY
- **架构图（HTML/SVG/PDF/PNG）必须同步重建**。用户会检查 HTML 架构图、PDF 输出和 PNG 截图是否全部更新，不能只更新 MD 文档
- **文档文件名也要对应升级**（如 V3→V4），不能旧文件名引用新内容
- 用户会主动检查文档是否更新，不要等用户指出后再补

## 交流提示

- 用户说"memory"指 Hermes Agent 持久化记忆（memory + user profile），不是系统 RAM
- 问"检查 memory"时先调 memory 工具查看条目，再用 terminal 看系统内存（如有需要）
- 用户说"帮我操作"时可直接执行，无需再确认权限
- 项目目录分类规则（work/personal/archive）优先放记忆而非 skill——操作规则/工作流入 skill，事实性约定入 memory

## 任务执行规则

- 用户批准计划后，严格按 TODO 列表顺序逐一执行，完成一步再做下一步
- 不要在第一步尚未完成验证时就跳到第二步
- 每步完成后更新 TODO 状态并通知用户进度
- 复杂任务先在 TODO 中列出所有步骤让用户确认，再启动执行

## 处理过程反馈偏好

用户反复强调过处理过程中必须有可见的进度反馈：
- 任何耗时操作（文件读取、API调用、后台处理）必须立即显示进度指示器（⏳/进度步骤/按钮文字变化）
- 按钮响应后必须立即变为禁用状态 + 文字改为操作说明（如「⏳ 分析中...」→「📤 上传分析中...」）
- 不要出现点击后无任何反应的状态——「无反映」是用户最反感的体验
- 过滤条件输入后，整个视图（统计卡+列表）应同步更新，清空过滤后恢复原始数据

## 输出偏好

- 数据源自查: 回答需注明数据来源（本地Doris/网络/知识库/文件）
- LI 数据必须标注是否本地且不出本地
- 分析报告使用纯文本 (.txt) 格式，不用 Markdown (.md)
  - 章节编号用 一、二、三...
  - 表格用空格对齐，不用 Markdown 表格语法
  - 数据在前，结论在后

- 输出语言：所有系统提示、欢迎界面、对话输出必须使用简体中文，仅必要英文术语（API 名称、协议名、命令等）保留原文
- 输出简洁、结构化，多用表格
- 迁移/交接类文档必须包含三部分：**最终交付物总览**（表格形式）、**完整操作步骤**（分步编号）、**注意点**（数据安全/平台差异/后续操作）
- 先本地验证，再远程操作
- 健康检查逐项列出细节，不笼统汇总
- 图表/可视化交付方式：MEDIA（图片）或 Chrome HTML
- 暗色主题数据可视化（matplotlib 暗色主题金融图+数据表）

### 系统健康看板格式偏好

当输出「Hermes 健康看板」或「系统概览」时，必须遵循以下格式约定（用户明确纠正过多次）：

**看板结构顺序（固定）：**
1. **📈 趋势** — 最顶部，展示系统负载/内存/磁盘/Cron 执行成功率等近期趋势数据
2. **📦 基础设施** — 主机名、系统版本、运行时间、CPU、内存、磁盘
3. **🔌 MCP 与服务** — MCP 服务列表、后台进程、Snap 服务
4. **🌐 Clash 机场** — Clash Verge 进程 + 代理端口 + 节点信息合并到一个区块（不拆分）
5. **⏱ Cron / Snap** — Hermes Cron 列表 + Snap 包
6. **📚 知识库与记忆** — 放在最后

**看板核心规则：**
- 始终使用终端文本格式（框线符号 `┌─┐│└┘` 和 emoji 图标），**不要**输出 Grafana 风格的 HTML/JSON 仪表盘（除非用户明确要求安装 Grafana）
- 相关概念必须合并：Clash Verge 的进程/端口/节点/规则集全部放在 **Clash 机场** 一个区块，不拆到 MCP 或其他区域
- 趋势区块必须基于实时数据（`uptime`、`free -h`、`df -h`、cron 执行统计），不可笼统描述
- Cron 区块中标注失败的推送任务及其原因

**列表示例格式（严格按此排版）：**
```
┌──────────────────────────────────────────────────────────┐
│                 Hermes 健康看板 · rhino01                 │
│                 2026-06-24 18:12 CST                      │
└──────────────────────────────────────────────────────────┘

  📈 趋势
  ───────────────────────────────────────────────────────────
  负载       0.97 / 1.30 / 1.39                         → 平稳
  Cron 执行  21 成功 / 3 失败 (87.5%)                    → 3 个均为推送侧
  ⚠ 微信 iLink 限流 (2), 飞书 shutdown (1)
```

### 报告数据来源标注（重要）

所有交付给用户的报告/分析文档，每个结论必须明确标注**数据出处**，不可笼统描述为\\\"分析得出\\\"。具体要求：

- **每个数据点/每个表格/每个章节**都必须写明来源：哪个PCAP文件、报文序号、解码工具、知识库条目
- **公网IP等关键发现必须列出具体报文清单**（PCAP名 + 报文序号 + SIP方法 + LIID），不能只说\\\"从解码结果中发现\\\"
- 推荐在文档末尾统一用「数据出处说明」表格汇总各章节来源
- \\\"来源：本地知识库\\\"\\\"来源：PCAP1解码 (a8b0, #9772)\\\"\\\"来源：ZTLIG日志 cid-xxx.txt\\\" 等标注要具体到文件/行/报文级别
- 涉及多个PCAP文件时，要区分标注不同文件的解码结果，合并统计时注明总量构成
- 这是用户反复纠正过的硬性要求，所有分析报告必须遵守

## "学习"工作流（用户提供内容→知识库沉淀）

用户说"学习"作为命令动词，后接待学内容（技术文章、协议文档、架构说明、图片等），触发以下流程：

### 五种输入形态

| 输入方式 | 示例 | Agent 动作 |
|---------|------|-----------|
| **直接粘贴内容** | 用户将文章/笔记粘贴到消息中 | 即时结构化整理 → 发问是否入库 |
| **文件路径（文本）** | "学习 ~/xxx/xxx.md" | 读文件后结构化整理 → 发问是否入库 |
| **文件路径（图片）** | "调用百炼 学习图片" | 调用百炼（DashScope qwen3-vl-plus）或 SiliconFlow VLM 批量分析图片 → 汇总分析结果 → 结构化入库 |
| **流程图（.drawio）** | "学习 TMC接入流程图.drawio" | 用 Python 解压 drawio 格式（base64 + zlib deflate -15 + URL decode）→ 提取节点和连线 → 结构化整理 → 入库 |
| **压缩包（.rar）** | "解压并学习RAR文件" | unrar x 解压 → libreoffice --headless --convert-to txt 批量转 → delegate_task 子agent批量阅读 → 汇总入库 |

### 陷阱

- **draw.io 文件是压缩 XML**：内部 `diagram` 标签内容是 base64 → zlib deflate 数据，需解压后再 URL decode 才得到完整 XML。单元格值在 `value` 属性中，连线在 `source`/`target` 属性中。不能直接用 `read_file` 或 `grep` 提取。

- **图片扩展名大小写敏感**：glob 匹配需同时包括 `.png` 和 `.PNG`，否则大写扩展名的图片会被漏掉。正确模式：`*.png *.jpg *.jpeg *.PNG *.JPG *.JPEG` 或直接用 `search_files` 先找出所有图片再统一处理。

- **百炼视觉分析调用**：API 端点 `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`，模型 `qwen3-vl-plus`，API key 从 `~/.hermes/.env` 中 `DASHSCOPE_API_KEY` 提取（终端 grep 显示 *** 但文件内是真实值）。批量处理时建议先遍历所有文件编码打包，再逐张发送请求，结果先存 JSON 再统一入库。

- **RAR 中的 .doc 文件**：LibreOffice 可能缺 Java 运行环境（`Warning: failed to launch javaldx`），但不影响 doc→txt 转换。转换后文件可能很大（>3MB），建议用 `delegate_task` 子agent批量阅读分析后再提炼汇总。

- **enzyme 已移除（2026-06-30）**：Enzyme 已被 Qdrant + kb-index 替代，所有技能文档已更新。

### 执行流程

1. **结构化整理** → 将用户提供的内容提炼为 tabular 摘要（表格/分类/关键变化对比）
2. **确认请求** → 询问用户是否存入知识库（"是否要将这些内容写入知识库？"）
3. **检查现有文件** → 搜索 `~/knowledge/` 对应分类目录下已有的关联文件，准备交叉引用
4. **写入知识库** → 按知识库规范写 markdown，包含：
   - 结构化标题层级
   - 表格/列表/对比呈现
   - `[[]]` wikilinks 关联已有笔记
   - 源信息/日期/分类元数据
   - 数据级别标注（LI 数据标 LEVEL 3 或 LEVEL 5）
5. **刷新语义索引** → `cd ~/knowledge && kb-index`

   > **注意**：`kb-index` 需要从环境变量提取 API key（`DASHSCOPE_API_KEY`），确保 `~/.hermes/.env` 中已配置。索引刷新通过 FTS5 + 语义嵌入双通道完成，刷新成功与否如实向用户汇报。

6. **如实汇报** → 成功写入 + 索引状态（刷新成功/失败均如实告知）

### 约定

- 用户说"好的"、"好"即确认授权，无需二次追问
- 知识笔记的数据来源必须在回答中说明：本地知识库 / 网络 / 学习材料 / 文件
- LI 相关数据必须标注数据级别（LEVEL 3 = 标准知识可外发，LEVEL 5 = 绝对秘密不出本地）
- 详细流程参见 `knowledge-closed-loop` 技能的 Step 2-6 和知识口述采集模式

### 极低颗粒度逐段分享模式（厂商文档特有）

当用户通过「学习」命令分享厂商 LI 文档（如 ZTE HI / Utimaco RAI）时，呈现独特的碎片化交互模式：

#### 分享行为信号

- **单词级颗粒度**：用户可能一次只发一个单词或短语（如 `unchanged`、`characters`、`?`），每收到一次就立即更新对应字段
- **`?` 的含义**：单独发 `?` 通常表示"继续"、"确认"或"是的"（非疑问）。此时确认当前字段已更新并等待下一段，不要追问"你要什么"
- **空消息**：用户可能发送空内容。不要报错或追问——保持等待或简洁确认
- **页码信号**：用户用 `83 March 21 ?? Copyright Utimaco TS GmbH` 格式表明文档翻页。准备好接收新内容即可，不需要问"是否还有更多"
- **文档引用**：用户单独发 `Utimaco LIMS — Remote Administration Interface (RAI) v16.1` 是引用当前文档，不是新内容。简洁回应即可
- **重复文档**：同一份文档发了两次，立即告知"这篇文档与之前解析的是同一份"，不要重复入库

#### 参数分享与更新规则

- **先简后详的双阶段模式**：用户先一次性分享命令的语法（Syntax）——所有参数名和结构（如 `tmod tno_id=TargetNoId [liid=LiId] [net=Networks] ...`），然后进入参数描述阶段，逐词分享每个参数的原文英文描述。**即使你已经有了整段参数列表，仍需逐词等待用户分享描述**——用户在核对原文措辞，而不是教你参数含义
- **多轮示例修正**：命令示例分多轮完成——先发核心部分（`tmod tno_id=2 net=GSM,GPRS mc_data=42`），下一轮补充 `doo=20020501`。不要催促一次性贴完
- **状态码逐码分享**：用户逐个数字分享（`0` → `100-199` → `201, 202` → `250, 251` → `310-339`...）。每次收到就更新表格。不要提前补全其他命令已有的码值——不同命令可能有独有子集
- **参数描述逐词复核**：mc_xxx 参数描述分 3-4 段完成：第一段到 `If the argument is not present, the existing MC assignment is`，第二段到 `unchanged. Otherwise, McVoice must be the Id of an existing`，第三段到 `monitoring center, and will replace the existing voice MC.`，最后发 `?` 确认。每次立即 patch，不要提前合并
- **片段式语法响应**：用户只发一个词（`Syntax`、`Arguments`、`Remarks`）表示要分享该章节。添加章节标题后等待具体内容，不要提前推断

#### Patch 纪律

- **必须精确匹配原文英文措辞**，不接受概括、简写或翻译替代
- 即使用户只发一个词（如 `unchanged`），立即更新对应字段的特定词句并等待继续——不要"我帮你一次改完"或跳步
- 如果用 patch 复制了某行（如 262 号状态码出现了两行），必须立即修复
- **补丁匹配失败时**：先 `read_file` 查看目标区域的准确内容，确认上下文后再 patch。如果报 `Found N matches`，说明需要更具体的上下文（添加前后行；如果 `| 0 | 成功 |` 匹配到三个命令的表，用前一行含命令名的标识来区分）
- **所有 mc_xxx 参数共享语义**：不传保留、传入替换、值必须是已存在 MC ID。但每个参数仍有独特描述，逐字等待用户分享
- **状态码优先引用已有文档**，仅补充当前命令特有的码值
- 英文原文中的拼写错误（如文档中的 `configuered`）保留原样

## 技能文档标准

- 技能文档应完整，包含命令、诊断步骤、速查表
- Skill 导入时不缩写
- `.md` 用 Typora 打开，`.docx` 用 LibreOffice
- PDF 输出验证通过后交付，md→PDF 用 xelatex
